# 编排器 A/B 基准测试计划 — 详细回顾

[English](benchmarks.md) | [한국어](benchmarks.ko.md) | **简体中文**

本扩展内置的提示栈（worker 基础提示、调度策略、模型校准）不是一次写成的 —
它经过四轮对抗性 A/B 迭代调优。本文是完整回顾：每一轮测了什么、每个裁决背后的
证据、每一轮产出了哪条校准规则，以及整个计划关于"缩小 frontier 编排器与校准后的
非 frontier 编排器之间差距"教会了我们什么。

## 协议

每一轮都遵循同一形态：

1. **两个全新工作区，同一份任务书。** 完全相同的构建提示交给两个 Claude Code
   会话：一个由 frontier 层级的主模型编排，另一个由 `claude-opus-4-8` 编排。
   两侧使用相同的 worker 池、相同的调度策略、相同的 MCP 配置。worker 完全相同 —
   只有编排器不同，因此任何质量差异都可归因于编排本身：契约设计、验证攻击性、
   以及对评审发现的判断。
2. **盲评、基于实际执行的评分。** 第三个全新会话收到两个以中性名称命名的文件夹和
   一份 100 分制评分表。每一分都必须有裁判自己执行的命令作为依据 — 对两个构建
   发射完全相同的探针集（恶意输入、并发竞争、`kill -9` 恢复、文档与行为的一致性
   检查，视领域而定）。任务书留白的决策，按**各项目自己钉死的契约**来评判，
   而不是按另一个项目的选择。
3. **恰好一条建议。** 裁判必须以单条杠杆最高的校准补充作结。该规则只并入非
   frontier 的校准提示（除非 frontier 一侧也失败，否则绝不进入层级中立的静态
   策略），然后跑下一轮。

这个强制机制很重要：每轮一条建议防止校准膨胀成没人遵守的清单，也让每一轮可以
被证伪 — 如果规则有效，该失败类别就必须不再复发。

## 结果一览

| 轮次 | 任务 | Frontier | 校准后的 Opus | 差距 | 产出规则 |
|---|---|---|---|---|---|
| 1 | 记账 REST API（绿地，R1–R7） | 96.5 | 85 | 11.5 | 结构化的未知之未知搜寻 |
| 2 | 同一领域，更强的对抗探针 | 96 | 85 | 11 | 复现类别的否决程序 |
| 3 | 同一领域，更严格的执行型裁判 | 90 | 81 | 9 | 缺失 ≠ 无效；事前验尸可追溯性 |
| 4 | 作业队列 CLI（状态/并发/时间）+ 中途变更请求 | 99 | 96 | **3** | 文档一致性关卡 |

分数**跨轮不可比** — 评分表和裁判严格度每轮都在收紧（第 3 轮的裁判给同等质量的
frontier 构建打了比第 2 轮低 6 分的分数）。信号在于*轮内差距*以及造成差距的
*缺陷种类*。

## 第 1 轮 — 基线：未校准的差距究竟长在哪里

**任务。** 从零构建记账 REST API + 最小前端：带验证的 POST、过滤列表、月度汇总、
DELETE 语义、防 XSS 前端、≥25 个测试、README（R1–R7）。

**裁决：96.5 : 85。** 此时校准侧只有通用运行规则（强制委派、外化计划、自主性、
基于证据的汇报）。11.5 分的差距几乎全部分解为**基础契约漏洞**：只查形态不查含义
的验证规则、契约从未定义的失败路径、没有任何任务提示钉死的边缘输入 — 每一个都是
worker *本可以*处理的事，但前提是契约有交代。frontier 编排器的契约在任何 worker
开始敲代码之前就把这些钉死了。

**采纳的规则。** 第一个大校准块：*结构化地搜寻未知之未知* — 冻结任何契约前先做
事前验尸（premortem）、自行生成领域陷阱清单（类别是普适的，条目因项目而异）、
在**值语义**而非表面形态的层面定义验证、为**规格之外的失败路径**定义明确行为、
在验证文件中记录至少五个恶意探针、并把 fresh-context 对抗性评审作为独立任务调度。

**收获。** 差距从来不是"Opus 代码写得差"。两侧的代码都是 worker 写的。差距在于
一个编排器*提前看见陷阱*并把它们编码进契约；另一个是从裁判那里才得知。这个诊断 —
变量是契约分辨率 — 决定了之后每一轮的方向。

## 第 2 轮 — 失败类别上移一层

**任务。** 同一领域，但裁判扎得更深：平台抛出的失败路径、货币表示的边缘、
以及对抗性评审发现的处理。

**裁决：96 : 85。** 第 1 轮的规则起效了 — 基础漏洞消失。校准侧这次在高一层失败：

- **平台抛出的错误不属于任何人的契约。** 让*框架本身*在应用验证运行之前就抛错的
  探针（中间件里的 URI 解码错误、请求体大小限制）产生了未处理或形态不一致的响应。
  契约定义了应用自身的失败路径（第 1 轮的教训），却没定义平台的。
- **货币输入契约的强度**和前端契约的具体性落后于 frontier 侧。
- 最致命的：一个真实的对抗性评审发现被**狭窄复现后否决**了 — 编排器只试了评审者
  字面上的那个探针，在自己的环境里看到通过，就关闭了发现。类别是真实存在的；
  只是那单个探针不具有代表性。

**采纳的规则。** 否决程序：要否决一个评审发现，必须复现**缺陷类别**而非评审者的
字面探针 — 亲自运行至少两个变体（不同操作/方法、不同输入位置或大小），记录在
验证文件中，并把平台/框架本身抛出的错误视为该类别的成员，在契约的失败路径清单中
显式钉死。

**收获。** 校准规则确实能消化失败类别 — 然后 frontier 模型会找到再上一层。第 2 轮
还精确定位了差距：在编排的三个杠杆（契约设计、验证攻击性、对发现的判断）中，
代价最高的失误是**判断**失败而非设计失败。判断是最难委派的东西。

## 第 3 轮 — 一个根因穿着两件外衣

**任务。** 同一领域；这轮裁判亲自执行一切（启动两侧服务器、相同的 curl 探针集、
对 git 历史和验证文件做过程取证），整体评分更严格。

**裁决：90 : 81。** 功能上两个构建都很强；五个因素决定胜负，其中两个决定性因素
共享同一个根：

- `GET /api/expenses?category=&month=` — 校准侧对空过滤值返回 **400
  invalid_month**；frontier 侧返回 200 与完整列表。浏览器表单例行序列化空字段；
  校准侧自己的前端只是靠从不发送空参数才幸免。这是 API 契约缺陷，不是风格选择。
- 不带 `month` 的 `GET /api/summary` — 校准侧把参数**设为必填**（400），于是
  "总支出汇总"根本不可能；frontier 侧返回全期聚合。
- 校准侧前端在服务端实现了分类过滤，却**从未在 UI 中暴露** — 自己契约表面之间的
  自相矛盾。
- 错误码分类漂移（按契约自己的词汇应为 `invalid_body` 的地方出现了
  `invalid_json`/`invalid_amount`）。
- 过程取证：frontier 侧留下了版本化契约（v1.0→v1.1）、两轮干净的、回馈进契约的
  修复循环、数字始终一致的验证文件；校准侧的验证文件残留着未更新的中间数字。

两个决定性功能缺陷都是**对空/缺失输入施加的过度严格** — "严谨"瞄准了错误的靶子。
最锋利的细节：校准编排器*自己的事前验尸*（第 8 项）恰恰标记了这个陷阱，而实现
还是让它复活了。发现存在过；它只是从未变成契约条款或测试。

**采纳的规则。** 这次是两条，因为这一轮同时暴露了缺陷*和*流程泄漏：(a) **缺失 ≠
无效** — 对每个可选输入，把"省略或为空 = 未请求该功能，按默认行为成功"钉为契约
默认值，严格验证只保留给存在但无效的值；(b) **事前验尸可追溯性** — 每个事前验尸
原因必须映射到契约条款，每个条款映射到探针；只识别而不钉死的陷阱保护不了任何东西。

**收获。** 到第 3 轮，校准侧的*事前验尸已经不错了* — 瓶颈从"看不见陷阱"移到了
"无法把看见的东西运进契约"。这是机械的、可修复的流程缺口，恰恰是校准规则能编码的
那一类。

## 第 4 轮 — 领域切换，功能平手

**任务。** 刻意不同：文件后端的**作业队列 CLI**（零运行时依赖），正面命中第 1–3 轮
从未触碰的陷阱类别 — 并发 worker 下的 exactly-once 认领、SIGKILL 崩溃恢复、状态机
迁移、退避计时、优先级/FIFO 排序 — 外加只在 Phase 1 完成后才交付的 **Phase 2 变更
请求**：作业依赖、破坏性的 prune 命令、以及打破默认行为的变更。这检验了绿地轮次
无法测量的两件事：校准规则能否跨领域迁移，以及编排器能否在不回归的前提下修订已
冻结的契约。

**裁决：99 : 96 — 双方功能缺陷为零。** 包括最难的探针在内，所有功能探针全部打平：

- exactly-once：20 个作业、两个竞争的 `work --concurrency 4` 进程、重复三次 —
  执行 20 次、零重复，双方皆然。
- SIGKILL 恢复：双方都**按自己文档化的语义**恢复了孤儿作业（一方把被中断的尝试
  记为 `interrupted`，另一方记为 `orphaned` 并对幸存子进程做进程组 kill — 都有效，
  都遵守了自己钉死的契约）。
- 恶意命令字符串（`$(…)`、反引号、引号、非 ASCII）在双方都无注入痕迹地往返；
  双方独立选择了 argv 数组存储 + `shell: false`。
- 退避计时、依赖级联（失败的父作业 → 传递性 blocked/skipped 的子作业）、prune
  安全性、破坏性变更迁移，双方都与各自契约一致。

全部 3 分的差距在于**最后一英里的严谨分配**：

- 校准侧的对抗性评审抓到了真正的 CRITICAL（prune 一个已成功的父作业会让依赖它的
  子作业悄悄变成孤儿）。修复是真实的 — 但它落在了契约文件和代码里，**没落在
  README 里**。最重要的安全行为对用户不可见了。frontier 侧交付了同样的保护，
  *并带有* README 里写明的提示 — 裁判的探针里逐字观察到了那句话。
- 校准侧 README 声称"所有环境变量在启动时验证，exit 1" — 被观察反驳（某个子命令
  下 `JOBQ_STALE_MS=abc` → exit 0；裁判在下结论前把这个断言横跨四个变量、三个
  子命令都探了一遍）。
- 同一种损坏记录条件，两个子命令给出了两套不同的警告措辞。

有两个细节值得诚实记录。第一，流程扣分是**对称的**：frontier 侧唯一丢的一分是
因为缺少显式的 risk→clause→probe 表 — 而那正是我们的校准*强制要求*非 frontier 侧
产出的工件（校准侧有，并因此受到表扬）。校准已不再是纯粹的纠偏；至少在一处它超过了
frontier 原生流程。我们刻意没有把它移植回去 — frontier 层级在最大自由下表现最佳，
一分评分是可接受的代价。第二，双方的 fresh-context **frontier-worker 对抗性评审
各自抓到了一个真实的跨特性 CRITICAL**，是 40+ 个单特性探针都漏掉的 — 升级阶梯的
价值此前只是文献中的假设，这次在同一轮的两侧都得到了确认。

**采纳的规则。** 文档一致性关卡：面向用户的文档是与代码处于同一验证关卡之下的契约
表面。任何改变行为的修复都要在同一次调度中更新文档（文档属于修复的文件所有权
清单），并且在最终汇报前跑一遍一致性检查：提取文档中每一条可测试的断言、对每条
执行探针、把断言与观察的对照记入验证文件。

**收获。** 功能差距是可以关闭的 — 它确实被关闭了，跨越了领域切换，且没有任何早期
失败类别复发。在 3 分的水平上剩下的是"最后一小时严谨指向哪里"，那是资源分配习惯，
不是能力差异。

## 跨轮分析

### 失败类别的阶梯

每一轮的决定性缺陷都比上一轮微妙一层：

| 轮次 | 类别 | 性质 |
|---|---|---|
| 1 | 契约漏洞（只验形态、未定义失败路径） | 设计 |
| 2 | 平台抛出的失败；狭窄复现后的否决 | 设计 + **判断** |
| 3 | 对缺失输入的过度严格；事前验尸→契约的泄漏 | 语义 + **流程** |
| 4 | 修复后的文档漂移；断言与观察的缺口 | **分配** |

这个演进本身就是闭环有效的最强证据：规则落地后没有任何类别复发，包括跨越第 4 轮
的领域切换 — 第 3 轮的"缺失 ≠ 无效"规则在一个它从未为之书写的 CLI 标志世界里
依然成立。

### 编排器无法委派的三个杠杆

每一轮的 worker 都完全相同，因此差距的每一分都能追溯到以下之一：

1. **契约设计** — worker 开工之前钉死了什么（第 1–3 轮）。
2. **验证攻击性** — 编排器亲自尝试打破什么、探针有多恶意（第 1、4 轮）。
3. **对发现的判断** — 当评审说"这坏了"、编排器必须裁决时发生什么（第 2、4 轮）。
   它被证明是错起来代价最高、也最难程序化的一环；否决规则和文档关卡归根结底都是
   判断的脚手架。

### 提示词做不到的，结构做到了

提示迭代在第 2–3 轮之间饱和（11.5 → 11 → 9）。第 4 轮塌缩到 3 分与结构性杠杆的
开启同时发生：

- **升级阶梯**（把 frontier 模型用作设计咨询和对抗性评审的 *worker*）：得到实验
  验证 — 双方的 frontier 评审都抓到了饱和探测漏掉的 CRITICAL。这与验证者扩展的
  文献一致（强验证者比更多生成更能抬升较弱的生成者），也正是扩展要在这个杠杆周围
  强制**计费防护**的原因：价值是真的，frontier 模型转为按量计费后，成本也是真的。
- **契约锦标赛**（2–3 份独立草案，裁判合并）：第 4 轮中两份独立草案选择了相反的
  认领架构（rename-as-claim vs 锁文件租约），在同样的探针下*双双*被证明正确 —
  证明锦标赛的价值在于漏洞覆盖的多样性，而不是挑出"正确的"设计。

### 放置纪律

本计划的所有规则都住在**由简报下发的校准**里（只提供给非 frontier 编排器），
从不进入静态的 CLAUDE.md 策略。数据确认了理由：frontier 层级原生处理了所有这些
情况，而 frontier 的性能在指令表面最小时保存得最好。静态策略只承载层级中立的
结构性事实（委派原则、工具机制、升级阶梯的经济学）。当赌注是金钱而非质量时，
规则干脆离开提示词、进入**服务器强制的代码**（计费防护）— 提示词可以被无视；
被拒绝的调度不能。

## 有效性威胁

- **每格 n = 1。** 这是带证伪判据（类别不得复发）的调优循环，不是受控研究。
  应把轨迹而非任何单一分数当作结论。
- **跨轮评分表漂移**是真实且有意的（每任裁判都扎得更深）；跨轮分数比较没有意义。
  轮内比较受相同探针集与盲评文件夹名保护。
- **裁判的模型家族。** 裁判也是 Claude 会话；若存在系统性同族偏差，它会在轮内
  对双方等同作用，但可能压缩或拉伸差距。
- **基准过拟合**正是第 4 轮更换领域的明确原因；规则守住了。一次领域切换是证据，
  不是证明。
- **成本不对称未被测量。** frontier 侧每次构建的配额消耗不在评分表内；本计划
  优化的是每轮质量，不是每美元质量。计费防护的存在正是因为这两者现在可能分道扬镳。
