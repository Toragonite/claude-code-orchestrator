# 오케스트레이터 A/B 벤치마크 프로그램 — 상세 회고

[English](benchmarks.md) | **한국어** | [简体中文](benchmarks.zh-CN.md)

이 익스텐션에 내장된 프롬프트 스택(워커 기본 프롬프트, 디스패치 정책, 모델 보정)은
한 번에 쓰인 것이 아니라 네 번의 적대적 A/B 라운드로 튜닝되었습니다. 이 문서는 그
전체 회고입니다: 각 라운드가 무엇을 측정했는지, 판정의 근거가 된 증거, 각 라운드가
만들어낸 보정 규칙, 그리고 프로그램 전체가 "프론티어 오케스트레이터와 보정된
비프론티어 오케스트레이터의 격차 좁히기"에 대해 가르쳐준 것.

## 프로토콜

모든 라운드는 같은 형태를 따랐습니다:

1. **두 개의 새 워크스페이스, 하나의 브리프.** 동일한 빌드 프롬프트가 두 Claude Code
   세션에 들어갑니다: 하나는 프론티어 티어 메인 모델이, 하나는 `claude-opus-4-8`이
   오케스트레이션. 워커 풀·디스패치 정책·MCP 구성은 동일합니다. 워커가 동일하므로
   품질 차이는 전부 오케스트레이션 — 계약 설계, 검증 공격성, 리뷰 발견에 대한 판단
   — 으로 귀속됩니다.
2. **블라인드, 실행 기반 판정.** 제3의 새 세션이 중립적인 폴더명으로 두 결과물과
   100점 루브릭을 받습니다. 모든 점수는 판정자가 직접 실행한 명령으로 뒷받침되어야
   하며, 동일한 프로브 세트가 양쪽에 발사됩니다(적대적 입력, 동시성 경쟁, `kill -9`
   복구, 문서-대-동작 검사 등 도메인이 요구하는 것). 브리프가 열어둔 결정은 상대
   프로젝트의 선택이 아니라 **각 프로젝트 자신이 고정한 계약** 기준으로 판정합니다.
3. **권고는 정확히 하나.** 판정자는 가장 레버리지 높은 보정 추가 항목 단 하나로
   끝맺어야 합니다. 그 규칙은 비프론티어 보정 프롬프트에만 반영되고(프론티어 쪽도
   실패한 경우가 아니면 티어 중립적인 정적 정책에는 넣지 않음) 다음 라운드가 돌아갑니다.

이 강제 장치가 중요합니다: 라운드당 권고 하나는 보정이 아무도 안 따르는 체크리스트로
비대해지는 것을 막고, 각 라운드를 반증 가능하게 만듭니다 — 규칙이 작동한다면 그 실패
클래스는 재발하지 않아야 합니다.

| 1 | 지출 추적 REST API (그린필드, R1–R7) | 96.5 | 85 | 11.5 | 구조화된 unknown-unknowns 사냥 |

점수는 **라운드 간 비교 불가**입니다 — 루브릭과 판정 엄격도가 매 라운드 강해졌습니다
(3라운드 판정자는 비슷한 품질의 프론티어 빌드를 2라운드보다 6점 낮게 채점). 신호는
*라운드 내 격차*와 그 격차를 만든 *결함의 종류*입니다.

## 라운드 1 — 베이스라인: 무보정 격차의 실체

**과제.** 지출 추적 REST API + 최소 프론트엔드를 처음부터: 검증된 POST, 필터 목록,
월별 요약, DELETE 의미론, XSS 안전 프론트, 테스트 ≥25, README (R1–R7).

**판정: 96.5 : 85.** 이 시점의 보정은 일반 운영 규칙(위임 강제, 외부화된 계획,
자율성, 근거 기반 보고)뿐이었습니다. 11.5점 격차는 거의 전부 **기본 계약 구멍**으로
분해됐습니다: 형태만 검사하고 의미는 검사하지 않는 검증 규칙, 계약이 정의하지 않은
실패 경로, 어떤 태스크 프롬프트도 고정하지 않은 엣지 입력 — 하나하나가 워커가
처리*할 수 있었던* 것들이지만, 계약이 시켰을 때만 그렇습니다. 프론티어
오케스트레이터의 계약은 워커가 타이핑을 시작하기 전에 이것들을 고정해뒀습니다.

**채택된 규칙.** 첫 대형 보정 블록: *unknown unknowns를 구조적으로 사냥하라* —
계약 동결 전 premortem, 스스로 생성하는 도메인 footgun 체크리스트(카테고리는 보편,
항목은 프로젝트별), 표면 형태가 아닌 **값 의미론** 수준의 검증 정의, **스펙 밖 실패
경로**의 명시적 동작, 검증 파일에 기록하는 적대 프로브 ≥5개, 그리고 별도 태스크로
디스패치하는 fresh-context 적대적 리뷰.

**교훈.** 격차는 "Opus가 코드를 더 못 쓴다"가 아니었습니다. 코드는 양쪽 다 워커가
썼습니다. 격차는 한쪽 오케스트레이터는 *함정을 미리 보고* 계약에 인코딩했고, 다른
쪽은 판정자에게서 알게 됐다는 것. 이 진단 — 변수는 계약 해상도다 — 이 이후 모든
라운드의 방향을 정했습니다.

## 라운드 2 — 실패 클래스가 한 단계 올라오다

**과제.** 같은 도메인이지만 판정이 더 깊이 찔렀습니다: 플랫폼이 던지는 실패 경로,
돈 표현의 엣지, 적대 리뷰 발견의 처리.

**판정: 96 : 85.** 1라운드 규칙은 작동했습니다 — 기본 구멍은 사라졌습니다. 보정된
쪽은 이제 한 단계 위에서 실패했습니다:

- **플랫폼이 던지는 에러는 누구의 계약도 아니었다.** 애플리케이션 검증이 돌기 전에
  *프레임워크 자체*가 던지게 만드는 프로브(미들웨어의 URI 디코딩 에러, 페이로드
  크기 제한)가 처리되지 않거나 형태가 제각각인 응답을 냈습니다. 계약은 앱 자신의
  실패 경로(1라운드의 교훈)는 정의했지만 플랫폼의 것은 아니었습니다.
- **돈 입력 계약의 강도**와 프론트엔드 계약 구체성이 프론티어 쪽에 뒤졌습니다.
- 가장 뼈아픈 것: 진짜 적대 리뷰 발견 하나가 **좁은 재현 후 기각**됐습니다 —
  오케스트레이터가 리뷰어의 문자 그대로의 프로브만 시도해보고, 자기 환경에서
  통과하자 발견을 닫았습니다. 클래스는 실재했고, 프로브 하나가 대표적이지 않았을
  뿐입니다.

**채택된 규칙.** dismissal 절차: 리뷰 발견을 기각하려면 리뷰어의 문자 그대로의
프로브가 아니라 **버그 클래스**를 재현해야 한다 — 최소 두 개의 변형(다른
연산/메서드, 다른 입력 위치나 크기)을 직접 실행해 검증 파일에 기록하고, 플랫폼/
프레임워크 자체가 던지는 에러를 클래스의 일원으로 취급해 계약의 실패 경로 목록에
명시적으로 고정할 것.

**교훈.** 보정 규칙은 실패 클래스를 실제로 소화합니다 — 그러면 프론티어 모델이 그
다음 층을 찾아냅니다. 라운드 2는 격차의 위치도 정밀하게 짚었습니다: 오케스트레이션의
세 레버(계약 설계, 검증 공격성, 발견에 대한 판단) 중 가장 비싼 실수는 설계가 아니라
**판단** 실패였습니다. 판단은 위임하기 가장 어려운 것입니다.

## 라운드 3 — 두 개의 탈을 쓴 하나의 근본 원인

**과제.** 같은 도메인; 이번 판정자는 모든 것을 직접 실행했고(양쪽 서버 기동, 동일
curl 프로브 세트, git 히스토리·검증 파일에 대한 프로세스 포렌식) 전반적으로 더
엄격하게 채점했습니다.

**판정: 90 : 81.** 기능적으로 양쪽 다 강했고, 다섯 요인이 승부를 갈랐는데 결정적인
두 개는 하나의 뿌리를 공유했습니다:

- `GET /api/expenses?category=&month=` — 보정된 빌드는 빈 필터 값에 **400
  invalid_month**를 반환; 프론티어 빌드는 200과 전체 목록을 반환. 브라우저 폼은 빈
  필드를 일상적으로 직렬화하는데, 보정된 쪽의 프론트엔드는 빈 param을 아예 안
  보내는 식으로만 살아남았습니다. 스타일 선택이 아니라 API 계약 결함입니다.
- `month` 없는 `GET /api/summary` — 보정된 빌드는 파라미터를 **필수화**(400)해서
  "전체 지출 요약" 자체가 불가능했고, 프론티어 빌드는 전 기간 집계를 반환했습니다.
- 보정된 쪽 프론트엔드는 카테고리 필터를 서버에는 구현하고 **UI에는 노출하지 않음**
  — 자기 계약 표면들 사이의 자기모순.
- 에러 코드 택소노미 표류 (계약 자신의 어휘로는 `invalid_body`여야 할 곳에
  `invalid_json`/`invalid_amount`).
- 프로세스 포렌식: 프론티어 쪽은 버전 관리된 계약(v1.0→v1.1), 계약에 되먹인 두 번의
  깔끔한 수정 사이클, 숫자가 일관된 검증 파일을 남겼고; 보정된 쪽 검증 파일에는
  갱신 안 된 중간 수치가 잔존했습니다.

결정적인 두 기능 결함 모두 **빈/생략 입력에 적용된 과잉 엄격성**이었습니다 —
"엄밀함"이 잘못된 과녁을 겨눴습니다. 가장 날카로운 디테일: 보정된 오케스트레이터
*자신의 premortem*(8번 항목)이 정확히 이 함정을 지목했는데, 구현이 그것을 되살렸다는
것. 발견은 존재했지만 계약 조항도 테스트도 되지 못했습니다.

**채택된 규칙.** 이번엔 두 개 — 라운드가 결함과 프로세스 누수를 *동시에* 드러냈기
때문입니다: (a) **absent ≠ invalid** — 모든 선택적 입력에 대해 "생략·빈 값 = 기능
미요청, 기본 동작으로 성공"을 계약 기본값으로 고정하고, 엄격 검증은 존재하되 무효인
값에만; (b) **premortem 추적성** — 모든 premortem 원인은 계약 조항으로, 모든 조항은
프로브로 매핑돼야 한다; 식별만 되고 고정되지 않은 함정은 아무것도 보호하지 않는다.

**교훈.** 라운드 3에 이르러 보정된 쪽의 *premortem은 좋아졌습니다* — 병목이 "함정을
못 본다"에서 "본 것을 계약까지 운반하지 못한다"로 이동했습니다. 이것은 기계적이고
고칠 수 있는 프로세스 갭이며, 정확히 보정 규칙이 인코딩할 수 있는 종류입니다.

## 라운드 4 — 도메인 전환, 기능적 동률

**과제.** 의도적으로 다르게: 파일 기반 **잡 큐 CLI**(런타임 의존성 0). 라운드 1–3이
건드리지 않은 footgun 카테고리를 정면으로 — 동시 워커 하의 exactly-once 클레임,
SIGKILL 크래시 복구, 상태 머신 전이, 백오프 타이밍, 우선순위/FIFO 순서 — 더해서
Phase 1 완료 후에만 전달되는 **Phase 2 변경 요청**: 잡 의존성, 파괴적 prune 명령,
기본 동작을 깨는 변경. 그린필드 라운드로는 측정 불가능했던 두 가지를 검사했습니다:
보정 규칙의 도메인 전이 여부, 그리고 얼린 계약을 회귀 없이 개정하는 능력.

**판정: 99 : 96 — 기능 결함 양쪽 0.** 가장 어려운 프로브를 포함해 모든 기능 프로브가
동률이었습니다:

- exactly-once: 잡 20개, `work --concurrency 4` 프로세스 2개 경쟁, 3회 반복 — 실행
  20회, 중복 0, 양쪽 모두.
- SIGKILL 복구: 양쪽 모두 고아 잡을 **자신이 문서화한 의미론대로** 복구(한쪽은
  중단된 attempt를 `interrupted`로, 다른 쪽은 살아남은 자식 프로세스를 프로세스
  그룹 kill과 함께 `orphaned`로 — 둘 다 유효하고, 둘 다 자기 계약을 지킴).
- 적대적 명령 문자열(`$(…)`, 백틱, 따옴표, 비ASCII)이 양쪽에서 주입 흔적 없이
  라운드트립; 양쪽 다 독립적으로 argv 배열 저장 + `shell: false`를 선택.
- 백오프 타이밍, 의존성 캐스케이드(실패한 부모 → 이행적으로 blocked/skipped),
  prune 안전성, 파괴적 변경 마이그레이션 전부 양쪽에서 자기 계약과 일치.

3점 전부가 **마지막 마일의 엄밀함 배분**이었습니다:

- 보정된 쪽의 적대 리뷰가 진짜 CRITICAL을 잡았습니다(성공한 부모를 prune하면 의존
  자식이 조용히 고아가 됨). 수정은 실재했지만 — 계약 파일과 코드에만 반영되고
  **README에는 반영되지 않았습니다.** 가장 중요한 안전 동작이 사용자에게 보이지
  않게 된 겁니다. 프론티어 쪽은 같은 보호를 README에 문서화된 안내와 *함께* 출시했고,
  판정자의 프로브에서 그 문구가 그대로 관측됐습니다.
- 보정된 README는 "모든 환경 변수는 시작 시 검증, exit 1"이라 주장했지만 관측이
  반증했습니다(한 서브커맨드에서 `JOBQ_STALE_MS=abc` → exit 0; 판정자는 이 주장을
  변수 4개·서브커맨드 3개에 걸쳐 찔러본 뒤에야 판정).
- 하나의 손상 레코드 조건에 두 서브커맨드가 서로 다른 경고 어휘를 냈습니다.

정직하게 기록할 두 가지. 첫째, 프로세스 감점은 **대칭**이었습니다: 프론티어 쪽은
명시적 risk→clause→probe 표가 없다는 이유로 유일한 1점을 잃었는데 — 그 표는 우리
보정이 비프론티어 쪽에 *의무화*한 바로 그 산출물입니다(보정된 쪽은 갖고 있었고
칭찬받았습니다). 보정은 더 이상 순수 교정이 아니며, 적어도 한 지점에서는 프론티어
네이티브 프로세스를 초과합니다. 우리는 이것을 프론티어 쪽에 역이식하지 않기로 했습니다
— 프론티어 티어는 최대 자유에서 최고 성능을 내고, 루브릭 1점은 그 수용된 대가입니다.
둘째, 양쪽의 fresh-context **프론티어 워커 적대 리뷰가 각각, 40+개의 단일 기능
프로브가 놓친 실제 크로스 피처 CRITICAL을 잡아냈습니다** — 문헌의 가설이었던
에스컬레이션 래더의 가치가 같은 라운드의 양쪽에서 확인된 겁니다.

**채택된 규칙.** 문서 정합 게이트: 사용자 문서는 코드와 같은 검증 게이트 아래의 계약
표면이다. 동작을 바꾸는 모든 수정은 같은 dispatch에서 문서를 갱신하고(문서는 수정의
파일 소유 목록에 포함), 최종 보고 전 정합 패스가 문서의 모든 검증 가능한 주장을
추출해 프로브를 실행하고 주장-대-관측을 검증 파일에 기록한다.

**교훈.** 기능 격차는 닫을 수 있습니다 — 실제로 닫혔고, 도메인 전환을 가로질러 이전
실패 클래스의 재발 없이 닫혔습니다. 3점 수준에서 남는 것은 "마지막 한 시간에
엄밀함이 어디를 향하는가"이며, 이는 능력 차이가 아니라 자원 배분 습관입니다.

## 횡단 분석

### 실패 클래스의 사다리

각 라운드의 결정적 결함은 직전보다 한 단계 미묘했습니다:

| 라운드 | 클래스 | 성격 |
|---|---|---|
| 1 | 계약 구멍 (형태만 검증, 미정의 실패 경로) | 설계 |
| 2 | 플랫폼 발생 실패; 좁은 재현 기각 | 설계 + **판단** |
| 3 | 빈 입력 과잉 엄격; premortem→계약 누수 | 의미론 + **프로세스** |
| 4 | 수정 후 문서 표류; 주장-대-관측 갭 | **배분** |

이 진행 자체가 루프가 작동한다는 가장 강한 증거입니다: 규칙이 반영된 후 재발한
클래스는 없으며, 라운드 4의 도메인 전환도 넘었습니다 — 라운드 3의 "absent ≠ invalid"
규칙은 그것이 쓰인 적 없는 CLI 플래그 세계에서도 유지됐습니다.

### 오케스트레이터가 위임할 수 없는 세 개의 레버

워커는 모든 라운드에서 동일했으므로, 격차의 모든 점수는 다음 중 하나로 귀속됩니다:

1. **계약 설계** — 워커가 시작하기 전에 무엇이 고정되는가 (라운드 1–3).
2. **검증 공격성** — 오케스트레이터가 직접 무엇을 부수려 시도하고, 프로브가 얼마나
   적대적인가 (라운드 1, 4).
3. **발견에 대한 판단** — 리뷰가 "이거 고장났다"고 할 때 오케스트레이터의 결정
   (라운드 2, 4). 가장 비싸게 틀리는 것으로, 그리고 절차화하기 가장 어려운 것으로
   판명됐습니다; dismissal 규칙과 문서 게이트는 결국 둘 다 판단의 발판입니다.

### 프롬프팅이 못 한 것을 구조가 했다

프롬프트 반복은 라운드 2–3 사이에서 포화했습니다 (11.5 → 11 → 9). 라운드 4의 3점
붕괴는 구조적 레버가 켜진 시점과 일치합니다:

- **에스컬레이션 래더** (프론티어 모델을 설계 자문·적대 리뷰용 *워커*로): 실험적으로
  검증 — 양쪽의 프론티어 리뷰가 포화 프로빙이 놓친 CRITICAL을 잡았습니다. 검증자
  스케일링 문헌과 정합하며(강한 검증자는 더 많은 생성보다 약한 생성자를 더 멀리
  끌어올린다), 익스텐션이 바로 이 레버 주위에 **과금 가드**를 강제하는 이유입니다:
  가치가 실재하고, 프론티어 모델이 종량 과금으로 바뀐 뒤에는 비용도 실재합니다.
- **계약 토너먼트** (독립 초안 2–3개, 판정 병합): 라운드 4에서 두 독립 초안이 정반대
  클레임 아키텍처(rename-as-claim vs lock-file-lease)를 골랐고 *둘 다* 같은 프로브
  아래 정확했습니다 — 토너먼트의 가치는 "올바른" 설계 고르기가 아니라 구멍 커버리지의
  다양성이라는 증거.

### 배치 규율

이 프로그램의 모든 규칙은 **브리핑으로 전달되는 보정**(비프론티어 오케스트레이터에게만
제공)에 살고, 정적 CLAUDE.md 정책에는 절대 넣지 않았습니다. 데이터가 확인해준 근거:
프론티어 티어는 이 케이스들 전부를 네이티브로 처리했고, 프론티어 성능은 지시 표면을
최소로 유지할 때 가장 잘 보존됩니다. 정적 정책은 티어 중립적인 구조적 사실(위임 원칙,
도구 역학, 에스컬레이션 래더의 경제학)만 담습니다. 품질이 아니라 돈이 걸린 곳에서는
규칙이 프롬프트를 아예 떠나 **서버 강제 코드**(과금 가드)로 들어갔습니다 — 프롬프트는
무시될 수 있지만, 거부된 dispatch는 무시될 수 없습니다.

## 유효성에 대한 위협

- **셀당 n = 1.** 이것은 반증 기준(클래스 재발 금지)이 있는 튜닝 루프이지 통제된
  연구가 아닙니다. 개별 점수가 아니라 궤적을 결과로 읽어야 합니다.
- **라운드 간 루브릭 표류**는 실재하고 의도적입니다(판정마다 더 깊이 찔렀음); 라운드
  간 점수 비교는 무의미합니다. 라운드 내 비교는 동일 프로브 세트와 블라인드 폴더명으로
  보호됩니다.
- **판정자 모델 패밀리.** 판정자도 Claude 세션이었습니다; 체계적 동일 패밀리 편향이
  있다면 라운드 내 양쪽에 동등하게 작용하겠지만 격차를 압축하거나 늘릴 수 있습니다.
- **벤치마크 과적합**은 라운드 4가 도메인을 바꾼 명시적 이유였고, 규칙은 유지됐습니다.
  한 번의 도메인 전환은 증거이지 증명이 아닙니다.
- **비용 비대칭은 미측정.** 프론티어 쪽의 빌드당 쿼터 소비는 루브릭에 없었습니다;
  프로그램은 라운드당 품질을 최적화했지 달러당 품질이 아닙니다. 과금 가드는 정확히
  이 둘이 이제 갈라질 수 있기 때문에 존재합니다.
