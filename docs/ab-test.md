# Fable vs Opus 오케스트레이터 A/B 테스트

같은 워크스페이스(`~/fable-test`)에서 메인 모델만 바꿔 동일 과제를 두 번 실행하고, 제3의 평가 세션이 채점한다.

## 실행 규칙 (공정성)

- 두 런 모두 **새 Claude Code 세션**에서 시작 (컨텍스트 오염 방지)
- 런 사이에 패널에서 `/model`로 메인 모델만 변경 (워커 구성·정책·익스텐션 버전 동일)
- 각 런의 시작/종료 시각을 기록 (벽시계 시간도 지표)
- 대시보드에서 각 런의 체크인 모델(`main session: ...`)이 의도한 모델인지 확인

## 런 프롬프트 (붙여넣기용)

Fable 런은 아래 그대로, Opus 런은 `run-fable`을 `run-opus`로 전부 바꿔서 사용.

```
[A/B 벤치마크 — 작업 폴더: run-fable]

이 워크스페이스 안에 run-fable/ 폴더를 만들고, 이후 모든 작업(파일 생성/수정/실행)을
그 폴더 안에서만 해줘. 폴더 밖 파일은 읽기만 허용.
시작하자마자 run-fable/ 에서 git init 하고, 의미 있는 단계마다 커밋해줘
(계약 설계 / 병렬 구현 결과 / 각 수정 사이클 / 최종). 커밋 이력이 채점 대상이다.

만들 것: "가계부(expense tracker)" 풀스택 미니앱
스택: Node 22 + TypeScript + Express(메모리 저장소), 바닐라 HTML/JS 프론트엔드, vitest

요구사항 (R1~R7 전부 채점 항목):
 R1. POST /api/expenses {amount, category, memo?, date?}
     - 검증: amount는 0보다 큰 숫자, category는 food|transport|leisure|etc 중 하나
     - 실패 시 400 + {error, message}
 R2. GET /api/expenses?category=&month=YYYY-MM — 두 필터의 조합 지원
 R3. GET /api/summary?month=YYYY-MM — 카테고리별 합계 + 총합
 R4. DELETE /api/expenses/:id — 없으면 404
 R5. 프론트엔드: 입력 폼 + 월 선택 + 지출 목록 + 카테고리별 합계 막대 그래프
     (차트 라이브러리 금지 — div/CSS로 직접, XSS 안전하게)
 R6. vitest 테스트 25개 이상 — 라우트별 정상/에러 케이스 포함
 R7. README.md — 실행 방법 + API 명세 표

진행 방식은 CLAUDE.md의 dispatch 정책을 그대로 따른다. 특히:
- 계약(타입/API 스펙/파일 소유권) 설계 → dispatch_tasks 병렬 위임 → 직접 검증
  (typecheck + vitest + 실서버 스모크) → 실패분만 재dispatch, 전부 통과까지 반복.
완료 보고에 반드시 포함: 요구사항별 충족 여부 표, dispatch 총 횟수와 수정 사이클 수,
직접 작성한 파일 목록(계약 파일 외에 있다면 사유), 남긴 한계.
```

## 평가 프롬프트 (두 런 완료 후, 새 세션에서)

```
[A/B 품질 평가 — 심판 역할]

run-fable/ 과 run-opus/ 는 서로 다른 오케스트레이터가 동일한 요구사항으로 만든
가계부 앱이다. 편견 없이 심사해줘. 절차:

1. 각 폴더에서 직접 npm install, npm run build(또는 typecheck), npm test를 실행하고
   실서버를 띄워 R1~R5를 curl로 스모크 테스트해라. 실행 결과가 채점의 근거다.
2. 아래 루브릭으로 각각 100점 만점 채점, 항목마다 근거 한 줄:
   - 기능 충족 (R1~R7 각 8점 = 56점): 실제 동작 확인 기반. 부분 동작은 부분 점수.
   - 코드 품질 (20점): 계약 일관성, 모듈 경계, 불필요한 복잡성/과잉 방어 코드 없음
   - 테스트 품질 (12점): 에러 케이스 커버, 테스트 독립성, 의미 있는 단언
   - 프로세스 (12점): git log 기반 — 커밋 단위의 논리성, 수정 사이클 횟수(적을수록 가점),
     완료 보고의 정확성(보고와 실제 코드의 일치 여부)
3. 동점 근처 항목은 두 구현의 diff를 직접 열어 비교해라.
4. 마지막에: 종합 점수표, 승자, 격차를 만든 요인 상위 3개, 그리고
   "오케스트레이터 보정 프롬프트가 다음에 더 보완해야 할 것" 1가지 제안.
```

## 함께 기록할 프로세스 지표

| 지표 | 어디서 |
|---|---|
| 벽시계 소요 시간 | 수동 기록 |
| dispatch 횟수 / 수정 사이클 | 완료 보고 + Dispatched Tasks 뷰 |
| 워커 토큰 사용량 | 대시보드 (런 전후 스냅샷) |
| 메인이 직접 구현한 파일 수 | 완료 보고 + git log 작성자 시점 대조 |
| orchestrator_briefing 호출 여부 | 대시보드 main session 표시 |
